# Технології захисту інформації

## Класифікація загроз, вразливостей та атак

### Загрози інформаційній безпеці

Загроза — потенційна можливість порушення конфіденційності, цілісності або доступності інформації.

**За джерелом:**
- **Зовнішні** — хакери, конкуренти, шкідливе ПЗ
- **Внутрішні** — інсайдери, помилки персоналу, збої обладнання

**За характером впливу:**
- **Природні** — стихійні лиха, відключення електроенергії
- **Техногенні** — відмова обладнання, програмні помилки
- **Антропогенні** — навмисні або ненавмисні дії людей

### Вразливості

Вразливість — слабке місце в системі, яке може бути використане загрозою.

- **Програмні** — помилки в коді, переповнення буфера, SQL-ін'єкції
- **Конфігураційні** — стандартні паролі, відкриті порти, надмірні привілеї
- **Архітектурні** — відсутність сегментації мережі, слабка автентифікація

**CVE (Common Vulnerabilities and Exposures)** — єдина база ідентифікаторів відомих вразливостей.

### Класифікація атак

**За активністю:**
- **Пасивні** — перехоплення (sniffing), аналіз трафіку
- **Активні** — MITM, DoS/DDoS, ін'єкції, підміна даних

**За вектором:**
- **Мережеві** — сканування, спуфінг, ARP-poisoning
- **На застосунки** — SQL-ін'єкція, XSS, CSRF
- **Соціальна інженерія** — фішинг, претекстинг

## Засоби захисту мережі

### Міжмережеві екрани (Firewall)

Контролюють вхідний та вихідний мережевий трафік на основі заданих правил.

**Типи фаєрволів:**

| Тип | Рівень OSI | Особливості |
|-----|-----------|-------------|
| Пакетний фільтр | L3-L4 | Аналізує заголовки пакетів (IP, порт, протокол) |
| Stateful Inspection | L4 | Відстежує стан TCP-з'єднань |
| Application Gateway (Proxy) | L7 | Проксіює трафік, глибокий аналіз вмісту |
| WAF (Web Application Firewall) | L7 | Захист веб-застосунків від SQL-ін'єкцій, XSS |
| NGFW (Next-Gen Firewall) | L3-L7 | Поєднує функції фаєрвола, IPS, антивірусу |

### IDS — система виявлення вторгнень

IDS (Intrusion Detection System) працює в режимі моніторингу та сповіщає адміністратора про підозрілу активність.

**Класифікація IDS:**
- **NIDS (Network IDS)** — аналізує мережевий трафік (Snort, Suricata)
- **HIDS (Host IDS)** — контролює активність на окремому хості (OSSEC, Tripwire)

**Методи виявлення:**
- **Сигнатурний** — порівняння з базою відомих атак; точний, але не виявляє zero-day
- **Аномальний** — профілювання нормальної поведінки; здатний виявити невідомі атаки, але дає більше хибних спрацьовувань

### IPS — система запобігання вторгненням

IPS (Intrusion Prevention System) встановлюється inline (у розрив трафіку) і може автоматично блокувати шкідливий трафік. Поєднує функції виявлення (як IDS) з можливістю активного реагування: скидання з'єднань, блокування IP-адрес.

## Управління доступом

### Active Directory (AD)

Active Directory Domain Services (AD DS) — служба каталогу Microsoft для централізованого управління мережевою інфраструктурою Windows.

**Основні компоненти:**
- **Домен** — логічна група об'єктів (користувачі, комп'ютери)
- **Контролер домену (DC)** — сервер, що зберігає базу AD та виконує автентифікацію
- **Організаційна одиниця (OU)** — контейнер для групування об'єктів і застосування політик
- **Групові політики (GPO)** — централізоване управління конфігурацією та безпекою

**Протокол автентифікації:** Kerberos (квитки, KDC, TGT, TGS).

### Списки контролю доступу (ACL)

ACL (Access Control List) — набір правил, що визначають дозволи на доступ до ресурсів.

**У файлових системах (NTFS, ext4):**
- Визначають права читання, запису, виконання для користувачів і груп
- DACL (Discretionary ACL) — власник ресурсу визначає доступ
- SACL (System ACL) — визначає аудит доступу

**У мережевому обладнанні:**
- Фільтрація трафіку на маршрутизаторах за IP-адресами, портами, протоколами
- Standard ACL — фільтрує лише за IP-адресою джерела
- Extended ACL — фільтрує за IP джерела/призначення, портами, протоколами

## Захист операційних систем

### Захист Windows

- **Windows Defender / Microsoft Defender** — вбудований антивірус і захист від загроз
- **Windows Firewall** — вбудований міжмережевий екран
- **BitLocker** — шифрування дисків
- **UAC (User Account Control)** — контроль підвищення привілеїв
- **Group Policy (GPO)** — політики паролів, блокування облікових записів, обмеження ПЗ
- **AppLocker / WDAC** — контроль запуску застосунків
- **Windows Event Log** — журнали подій безпеки (Security, System, Application)

### Захист Linux

- **SELinux / AppArmor** — обов'язковий контроль доступу (MAC)
- **iptables / nftables** — міжмережевий екран на рівні ядра
- **chmod / chown** — управління правами доступу до файлів (модель UGO: User, Group, Other)
- **sudo** — делегування привілеїв без надання повного root-доступу
- **auditd** — система аудиту подій ядра
- **fail2ban** — блокування IP після невдалих спроб входу
- **chroot / namespaces / cgroups** — ізоляція процесів

## Антивірусний захист

### Методи виявлення шкідливого ПЗ

- **Сигнатурний аналіз** — порівняння з базою відомих зразків (швидкий, але не виявляє нові загрози)
- **Евристичний аналіз** — аналіз коду на підозрілі конструкції та послідовності команд
- **Поведінковий аналіз (sandbox)** — запуск у ізольованому середовищі та спостереження за діями
- **Хмарний аналіз** — перевірка хешів файлів через хмарну базу загроз

### Класифікація шкідливого ПЗ

- **Вірус** — заражає файли, потребує носія
- **Хробак (worm)** — самостійно розповсюджується мережею
- **Троян** — маскується під легітимне ПЗ
- **Ransomware** — шифрує дані та вимагає викуп
- **Spyware** — збирає конфіденційну інформацію
- **Rootkit** — приховує присутність шкідливого ПЗ в системі

## Моніторинг подій безпеки (SIEM)

### Що таке SIEM

SIEM (Security Information and Event Management) — система збору, нормалізації, кореляції та аналізу подій безпеки з різних джерел у реальному часі.

### Функції SIEM

- **Збір логів** — агрегація журналів з фаєрволів, IDS/IPS, серверів, робочих станцій
- **Нормалізація** — приведення подій із різних джерел до єдиного формату
- **Кореляція** — виявлення зв'язків між подіями для ідентифікації інцидентів
- **Оповіщення** — сповіщення про критичні інциденти в реальному часі
- **Звітність** — формування звітів для аудиту та відповідності стандартам

### Приклади SIEM-систем

- Splunk, IBM QRadar, ArcSight, Elastic SIEM, Microsoft Sentinel

## Аналіз вразливостей

### Процес управління вразливостями

1. **Виявлення** — сканування інфраструктури автоматизованими засобами
2. **Оцінка** — визначення критичності за шкалою CVSS (0–10)
3. **Пріоритизація** — ранжування за рівнем ризику
4. **Усунення** — встановлення патчів, зміна конфігурації
5. **Верифікація** — повторне сканування для підтвердження усунення

### Інструменти сканування вразливостей

- **Nessus** — комерційний сканер, широка база CVE
- **OpenVAS** — відкрите рішення для сканування вразливостей
- **Nmap** — сканування портів і сервісів, базове виявлення вразливостей
- **Nikto** — сканер вразливостей веб-серверів

### Шкала CVSS

CVSS (Common Vulnerability Scoring System) — стандартна шкала оцінки критичності вразливостей:
- **0.0** — відсутня
- **0.1–3.9** — низька
- **4.0–6.9** — середня
- **7.0–8.9** — висока
- **9.0–10.0** — критична

## Захист мережевих протоколів

### IPSec

IPSec (Internet Protocol Security) — набір протоколів для захисту IP-трафіку на мережевому рівні (L3).

**Компоненти:**
- **AH (Authentication Header)** — автентифікація та цілісність (без шифрування)
- **ESP (Encapsulating Security Payload)** — шифрування + автентифікація
- **IKE (Internet Key Exchange)** — узгодження ключів і параметрів безпеки (SA)

**Режими роботи:**
- **Транспортний** — шифрує лише дані пакета, заголовок IP залишається
- **Тунельний** — шифрує весь пакет, додає новий IP-заголовок (використовується у VPN)

### SSL/TLS

SSL (Secure Sockets Layer) / TLS (Transport Layer Security) — протоколи захисту даних на транспортному рівні.

- TLS 1.2 та TLS 1.3 — сучасні версії (SSL 2.0/3.0 вважаються вразливими)
- Забезпечують: автентифікацію сервера (сертифікати X.509), шифрування каналу, цілісність даних
- Використовують гібридне шифрування: асиметричне для обміну ключами, симетричне для передачі даних

### HTTPS

HTTPS = HTTP + TLS. Забезпечує захищену передачу веб-трафіку.

- Стандартний порт: 443
- Використовує сертифікати X.509, видані центрами сертифікації (CA)
- Захищає від перехоплення та модифікації даних між клієнтом і сервером

### RADIUS

RADIUS (Remote Authentication Dial-In User Service) — протокол автентифікації, авторизації та обліку (AAA) для мережевого доступу.

- Централізована автентифікація користувачів при підключенні до мережі (Wi-Fi, VPN)
- Працює за моделлю клієнт-сервер (UDP, порти 1812/1813)
- Підтримує інтеграцію з Active Directory та LDAP

### VPN

VPN (Virtual Private Network) — технологія створення захищеного тунелю через незахищену мережу.

**Типи VPN:**
- **Site-to-Site** — з'єднання двох мереж (між офісами)
- **Remote Access** — підключення віддаленого користувача до корпоративної мережі

**Протоколи VPN:**
- **IPSec** — шифрування на L3, стандарт для site-to-site VPN
- **OpenVPN** — використовує TLS, працює через TCP/UDP
- **WireGuard** — сучасний протокол із мінімальним кодом і високою швидкістю
- **L2TP/IPSec** — тунелювання на L2 із шифруванням IPSec

## Моделі загроз

### STRIDE

STRIDE — модель класифікації загроз, розроблена Microsoft. Кожна літера відповідає типу загрози:

| Загроза | Опис | Порушує |
|---------|------|---------|
| **Spoofing** (Підміна) | Видавання себе за іншого користувача або систему | Автентифікацію |
| **Tampering** (Втручання) | Несанкціонована модифікація даних | Цілісність |
| **Repudiation** (Відмова від авторства) | Заперечення виконаних дій | Неспростовність |
| **Information Disclosure** (Розголошення інформації) | Несанкціонований доступ до конфіденційних даних | Конфіденційність |
| **Denial of Service** (Відмова в обслуговуванні) | Блокування доступу до ресурсів | Доступність |
| **Elevation of Privilege** (Підвищення привілеїв) | Отримання прав, які не були надані | Авторизацію |

STRIDE використовується на етапі проєктування для систематичного аналізу потенційних загроз кожному компоненту системи.

### Cyber Kill Chain

Cyber Kill Chain — модель, розроблена Lockheed Martin, що описує 7 послідовних фаз кібератаки:

1. **Reconnaissance (Розвідка)** — збір інформації про ціль: IP-адреси, доменні імена, email-адреси, відкриті порти, вразливості
2. **Weaponization (Озброєння)** — створення шкідливого навантаження (exploit + payload), наприклад троян у PDF-документі
3. **Delivery (Доставка)** — передача шкідливого навантаження жертві через email, веб-сайт, USB тощо
4. **Exploitation (Експлуатація)** — активація вразливості на системі жертви для виконання шкідливого коду
5. **Installation (Встановлення)** — інсталяція бекдору або іншого шкідливого ПЗ для збереження доступу
6. **Command & Control (C2)** — встановлення каналу керування між зламаною системою та атакуючим
7. **Actions on Objectives (Дії на цілі)** — виконання кінцевої мети: крадіжка даних, шифрування, руйнування

Модель дозволяє захисникам виявити та зупинити атаку на будь-якій з фаз — чим раніше, тим краще.

### MITRE ATT&CK

MITRE ATT&CK (Adversarial Tactics, Techniques, and Common Knowledge) — публічна база знань про тактики та техніки, які використовують реальні зловмисники.

**Структура:**
- **Тактики** — стратегічні цілі атакуючого (Initial Access, Execution, Persistence, Privilege Escalation, Defense Evasion, Credential Access, Discovery, Lateral Movement, Collection, Exfiltration, Command and Control, Impact)
- **Техніки** — конкретні методи досягнення тактичних цілей (наприклад, Phishing — техніка для тактики Initial Access)
- **Підтехніки** — деталізація технік (наприклад, Spearphishing Attachment — підтехніка Phishing)

**Використання:**
- **Threat Hunting** — проактивний пошук ознак компрометації на основі відомих технік
- **Red/Blue Teaming** — оцінка та покращення захисту за матрицею ATT&CK
- **Gap Analysis** — виявлення прогалин у покритті засобів захисту
- **Incident Response** — класифікація дій атакуючого під час розслідування інцидентів

## Zero Trust Architecture

Zero Trust — архітектурний підхід до безпеки, заснований на принципі **"ніколи не довіряй, завжди перевіряй"** (never trust, always verify).

**Ключові принципи:**
- **Жоден суб'єкт не є довіреним за замовчуванням** — незалежно від того, чи знаходиться він усередині або ззовні периметра мережі
- **Мікросегментація** — поділ мережі на малі ізольовані зони з окремими політиками доступу для кожної зони
- **Перевірка кожного запиту** — кожен запит до ресурсу проходить автентифікацію та авторизацію, навіть якщо попередній запит був дозволений
- **Least Privilege (Мінімальні привілеї)** — користувачі та системи отримують лише мінімально необхідний рівень доступу
- **Continuous Verification (Безперервна верифікація)** — постійна перевірка стану пристрою, поведінки користувача та контексту запиту протягом усієї сесії

**Компоненти Zero Trust:**
- Сильна автентифікація (MFA)
- Контроль доступу на основі політик (Policy Engine)
- Моніторинг і аналітика поведінки
- Шифрування всіх комунікацій
- Інвентаризація та класифікація активів

Zero Trust замінює традиційну модель "периметрового захисту", де внутрішня мережа вважалася безпечною.

## DLP (Data Loss Prevention)

DLP (Data Loss Prevention / Запобігання витоку даних) — технологія та набір політик для виявлення та запобігання несанкціонованій передачі конфіденційних даних за межі організації.

### Типи DLP

| Тип | Опис |
|-----|------|
| **Network DLP** | Контролює дані в русі (data in motion): моніторинг email, веб-трафіку, FTP, месенджерів |
| **Endpoint DLP** | Контролює дані на кінцевих пристроях (data in use): копіювання на USB, друк, скріншоти, буфер обміну |
| **Cloud DLP** | Контролює дані в хмарних сервісах: Google Drive, OneDrive, SaaS-застосунки |

### Методи виявлення конфіденційних даних

- **За контентом (Content Inspection)** — аналіз вмісту файлів та повідомлень на наявність конфіденційної інформації (номери карток, паспортні дані)
- **За контекстом (Context-based)** — аналіз метаданих: хто відправляє, кому, через який канал, в який час
- **За регулярними виразами (Regex)** — пошук даних за шаблонами (наприклад, формат ІПН, номер банківської картки, email-адреси)
- **Цифрові відбитки (Fingerprinting)** — створення хешів конфіденційних документів для їх подальшого виявлення
- **Машинне навчання** — класифікація документів на основі навчених моделей

## MFA (Multi-Factor Authentication)

MFA (Multi-Factor Authentication / Багатофакторна автентифікація) — метод автентифікації, що вимагає підтвердження особи за допомогою двох або більше незалежних факторів.

### Три фактори автентифікації

| Фактор | Тип | Приклади |
|--------|-----|----------|
| **Знання** (Something you know) | Те, що користувач знає | Пароль, PIN-код, секретне питання |
| **Володіння** (Something you have) | Те, що користувач має | Смартфон, апаратний токен, смарт-карта |
| **Біометрія** (Something you are) | Те, чим користувач є | Відбиток пальця, розпізнавання обличчя, сітківка ока |

Використання двох факторів з однієї категорії (наприклад, два паролі) **не є** MFA.

### Технології MFA

- **TOTP (Time-based One-Time Password)** — одноразовий код, що генерується на основі спільного секрету та поточного часу (Google Authenticator, Authy). Код змінюється кожні 30 секунд
- **FIDO2/WebAuthn** — стандарт безпарольної автентифікації з використанням апаратних ключів (YubiKey) або біометрії пристрою. Базується на криптографії з відкритим ключем, стійкий до фішингу
- **SMS/Email OTP** — одноразовий код через SMS або email (вважається менш безпечним через можливість перехоплення)
- **Push-сповіщення** — підтвердження входу через push-повідомлення в мобільному застосунку

## Honeypot / Honeynet

### Honeypot

Honeypot (пастка) — спеціально створена система-приманка, що імітує реальний ресурс (сервер, сервіс, базу даних) для привертання уваги зловмисників.

**Призначення:**
- Виявлення спроб несанкціонованого доступу
- Збір інформації про тактики, техніки та інструменти атакуючих
- Раннє попередження про атаку
- Відволікання зловмисників від реальних ресурсів

### Типи Honeypot

| Тип | Опис | Переваги | Недоліки |
|-----|------|----------|----------|
| **Low-interaction** | Емулює лише окремі сервіси (порти, банери) | Легко розгорнути, мінімальний ризик | Обмежений збір даних, легко розпізнати |
| **High-interaction** | Повноцінна реальна система з ОС та сервісами | Детальний збір даних, складно розпізнати | Ризик компрометації, складне обслуговування |

### Honeynet

Honeynet — мережа з кількох honeypot-систем, що імітує повноцінну інфраструктуру організації. Дозволяє відстежувати lateral movement (бокове переміщення) атакуючого та вивчати повний цикл атаки.

## PAM (Privileged Access Management)

PAM (Privileged Access Management / Управління привілейованим доступом) — набір технологій та практик для контролю, моніторингу та захисту доступу привілейованих облікових записів (root, administrator, service accounts).

**Основні функції:**
- **Сховище паролів (Password Vault)** — централізоване зберігання та автоматична ротація паролів привілейованих облікових записів
- **Сесійний запис (Session Recording)** — запис усіх дій адміністраторів під час привілейованих сесій (відео, команди, натискання клавіш) для подальшого аудиту
- **Just-in-Time Access (JIT)** — надання привілейованого доступу лише на обмежений час для конкретного завдання, після чого доступ автоматично відкликається
- **Контроль команд** — обмеження переліку дозволених команд для привілейованих користувачів
- **Аналіз поведінки** — виявлення аномальної активності привілейованих сесій

**Приклади рішень:** CyberArk, BeyondTrust, Thycotic, HashiCorp Vault.

## NAC (Network Access Control)

NAC (Network Access Control / Контроль мережевого доступу) — технологія, що перевіряє стан пристрою та автентифікує користувача **перед** наданням доступу до мережі.

**Принцип роботи:**
1. Пристрій підключається до мережі
2. NAC-система перевіряє: автентифікацію користувача, наявність антивірусу, актуальність оновлень ОС, відповідність політикам безпеки
3. За результатами перевірки пристрій отримує: повний доступ, обмежений доступ або відмову з переспрямуванням у карантинну мережу

**802.1X — стандарт автентифікації у мережі:**
- Використовує три компоненти: Supplicant (клієнт), Authenticator (комутатор/точка доступу), Authentication Server (RADIUS)
- Контролює доступ на рівні порту комутатора або бездротового з'єднання
- До успішної автентифікації пристрій має доступ лише до сервера автентифікації

## Backup стратегії

### Правило 3-2-1

Золоте правило резервного копіювання:
- **3** копії даних (1 основна + 2 резервні)
- **2** різних типи носіїв (наприклад, HDD + хмара)
- **1** копія зберігається в іншому фізичному місці (offsite)

### Типи резервного копіювання

| Тип | Що копіює | Швидкість створення | Швидкість відновлення | Обсяг |
|-----|-----------|--------------------|-----------------------|-------|
| **Повний (Full)** | Усі дані повністю | Найповільніше | Найшвидше | Найбільший |
| **Інкрементальний (Incremental)** | Лише зміни з моменту **останнього будь-якого** бекапу | Найшвидше | Найповільніше (потрібен повний + усі інкременти) | Найменший |
| **Диференціальний (Differential)** | Усі зміни з моменту **останнього повного** бекапу | Середнє | Середнє (потрібен повний + останній диференціальний) | Середній |

**Ключова різниця:** інкрементальний бекап зберігає зміни відносно попереднього бекапу (будь-якого типу), а диференціальний — завжди відносно останнього повного бекапу.

### RPO та RTO

- **RPO (Recovery Point Objective)** — максимально допустимий обсяг втрати даних у часі. Визначає, як часто потрібно робити бекапи. Наприклад, RPO = 1 година означає, що при збої можна втратити максимум 1 годину даних
- **RTO (Recovery Time Objective)** — максимально допустимий час відновлення системи після збою. Визначає, як швидко потрібно відновити роботу. Наприклад, RTO = 4 години означає, що система має запрацювати протягом 4 годин

## Patch Management

Patch Management (Управління оновленнями) — систематичний процес виявлення, тестування та встановлення оновлень безпеки для програмного забезпечення та операційних систем.

### Процес Patch Management

1. **Виявлення (Identification)** — моніторинг нових вразливостей та випущених патчів від виробників (CVE, бюлетені безпеки)
2. **Оцінка та пріоритизація** — визначення критичності патча за шкалою CVSS та оцінка впливу на інфраструктуру
3. **Тестування (Testing)** — перевірка патча в тестовому середовищі на сумісність і відсутність побічних ефектів
4. **Розгортання (Deployment)** — поетапне встановлення патча на продуктивні системи (спочатку пілотна група, потім масове розгортання)
5. **Верифікація** — підтвердження успішного встановлення та перевірка працездатності систем

**Пріоритизація за CVSS:**
- Критичні (9.0–10.0) — негайне встановлення (протягом 24–48 годин)
- Високі (7.0–8.9) — встановлення протягом тижня
- Середні (4.0–6.9) — встановлення протягом місяця
- Низькі (0.1–3.9) — встановлення під час планового обслуговування

## Класифікація інформації в Україні

Відповідно до законодавства України, інформація класифікується за рівнем обмеження доступу:

| Рівень | Гриф | Опис |
|--------|------|------|
| **Відкрита інформація** | Без грифу | Загальнодоступна інформація, поширення якої не обмежується |
| **Конфіденційна інформація** | «Конфіденційно» | Інформація з обмеженим доступом, розголошення якої може завдати шкоди інтересам особи або організації |
| **Таємна інформація** | «Таємно» | Державна таємниця, розголошення якої може завдати шкоди національній безпеці |
| **Цілком таємна інформація** | «Цілком таємно» | Найвищий рівень секретності, розголошення може завдати виняткової шкоди національній безпеці |

**Нормативна база:**
- Закон України «Про інформацію»
- Закон України «Про державну таємницю»
- Закон України «Про захист персональних даних»

Доступ до таємної та цілком таємної інформації надається лише особам, що мають відповідний допуск до державної таємниці.

## Хмарна безпека

### Моделі хмарних сервісів

| Модель | Опис | Приклади |
|--------|------|----------|
| **IaaS (Infrastructure as a Service)** | Надання віртуальної інфраструктури: сервери, мережі, сховища | AWS EC2, Azure VMs, Google Compute Engine |
| **PaaS (Platform as a Service)** | Надання платформи для розробки та розгортання застосунків | Azure App Service, Google App Engine, Heroku |
| **SaaS (Software as a Service)** | Надання готового ПЗ через Інтернет | Microsoft 365, Google Workspace, Salesforce |

### Shared Responsibility Model (Модель спільної відповідальності)

Відповідальність за безпеку розподіляється між хмарним провайдером та клієнтом:

| Компонент | IaaS | PaaS | SaaS |
|-----------|------|------|------|
| Фізична інфраструктура | Провайдер | Провайдер | Провайдер |
| Мережа | Провайдер | Провайдер | Провайдер |
| Гіпервізор / Віртуалізація | Провайдер | Провайдер | Провайдер |
| Операційна система | **Клієнт** | Провайдер | Провайдер |
| Middleware / Runtime | **Клієнт** | Провайдер | Провайдер |
| Застосунки | **Клієнт** | **Клієнт** | Провайдер |
| Дані | **Клієнт** | **Клієнт** | **Клієнт** |
| Управління доступом | **Клієнт** | **Клієнт** | **Клієнт** |

**Правило:** дані та управління доступом **завжди** залишаються відповідальністю клієнта, незалежно від моделі.

## Соціальна інженерія

Соціальна інженерія — метод маніпулювання людьми з метою отримання конфіденційної інформації або виконання певних дій, що оминає технічні засоби захисту.

### Типи атак соціальної інженерії

| Тип | Канал | Опис |
|-----|-------|------|
| **Фішинг (Phishing)** | Email | Масові розсилки підроблених листів від імені відомих організацій (банки, поштові сервіси) зі шкідливими посиланнями або вкладеннями |
| **Спірфішинг (Spear Phishing)** | Email | Цілеспрямований фішинг на конкретну особу або організацію з використанням персоналізованої інформації |
| **Вішинг (Vishing)** | Телефон | Голосовий фішинг — шахрайські дзвінки з метою виманити конфіденційні дані (наприклад, дзвінок "з банку") |
| **Смішинг (Smishing)** | SMS | Фішинг через SMS-повідомлення зі шкідливими посиланнями |
| **Претекстинг (Pretexting)** | Будь-який | Створення вигаданого сценарію (претексту) для отримання інформації: видавання себе за колегу, техпідтримку, керівника |
| **Baiting (Наживка)** | Фізичний / Цифровий | Підкидання заражених USB-носіїв або пропозиція безкоштовного завантаження з шкідливим ПЗ |
| **Tailgating (Прохід слідом)** | Фізичний | Проникнення до захищеного приміщення слідом за авторизованою особою без власної автентифікації |
| **Quid pro quo (Послуга за послугу)** | Будь-який | Пропозиція допомоги або послуги в обмін на інформацію: наприклад, "техпідтримка" пропонує виправити проблему, якщо користувач надасть пароль |

### Методи захисту від соціальної інженерії

- Регулярне навчання та тренінги з підвищення обізнаності (Security Awareness Training)
- Симуляція фішингових атак для перевірки пильності персоналу
- Політики верифікації особи при запитах конфіденційної інформації
- Багатофакторна автентифікація (MFA)

## IPsec детально

### AH vs ESP

| Характеристика | AH (Authentication Header) | ESP (Encapsulating Security Payload) |
|----------------|---------------------------|--------------------------------------|
| Автентифікація | Так (включаючи IP-заголовок) | Так (лише payload) |
| Шифрування | **Ні** | **Так** |
| Цілісність | Так | Так |
| Протокол IP | 51 | 50 |
| Сумісність з NAT | **Ні** (бо захищає IP-заголовок) | Так (з NAT-T, UDP 4500) |

На практиці **ESP використовується значно частіше**, оскільки забезпечує і шифрування, і автентифікацію. AH використовується рідко через несумісність з NAT.

### Transport vs Tunnel Mode

| Характеристика | Transport Mode | Tunnel Mode |
|----------------|---------------|-------------|
| Що захищає | Лише payload (дані) пакета | Весь оригінальний IP-пакет |
| IP-заголовок | Оригінальний залишається | Додається новий зовнішній IP-заголовок |
| Типове застосування | Зв'язок host-to-host | Site-to-Site VPN, Remote Access VPN |
| Приховування адрес | Ні — видно оригінальні IP-адреси | Так — оригінальні IP приховані |

### IKE (Internet Key Exchange)

IKE — протокол для автоматичного узгодження параметрів безпеки та обміну ключами. Складається з двох фаз.

**IKE Phase 1 — встановлення захищеного каналу (ISAKMP SA):**

Мета: створити безпечний канал для подальших переговорів Phase 2.

Два режими:
- **Main Mode (Основний)** — 6 повідомлень, забезпечує захист ідентичності сторін (Identity Protection). Більш безпечний
- **Aggressive Mode (Агресивний)** — 3 повідомлення, швидший, але ідентичність передається відкрито. Вразливий до перехоплення

Узгоджуються: алгоритм шифрування, алгоритм хешування, метод автентифікації, група Diffie-Hellman, час життя SA.

**IKE Phase 2 — узгодження IPsec SA (Quick Mode):**

Мета: створити IPsec SA для захисту реального трафіку.

- Використовує захищений канал, створений у Phase 1
- Узгоджує параметри ESP/AH: алгоритми шифрування та автентифікації
- Підтримує **Perfect Forward Secrecy (PFS)** — генерація нових ключів для кожної сесії
- Створює **дві односпрямовані SA** (одна для кожного напрямку)

### SA та SPI

- **SA (Security Association)** — набір узгоджених параметрів безпеки між двома сторонами: алгоритми, ключі, режим роботи, час життя. SA є односпрямованою — для двостороннього зв'язку потрібні дві SA
- **SPI (Security Parameters Index)** — 32-бітний ідентифікатор, що міститься в заголовку AH/ESP і однозначно визначає, яка SA застосовується до конкретного пакета. Приймач використовує SPI для вибору правильних параметрів дешифрування
