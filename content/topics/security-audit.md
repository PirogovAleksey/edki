# Аудит інформаційної безпеки

## Поняття, мета, завдання та принципи аудиту ІБ

**Аудит інформаційної безпеки** — систематичний, незалежний і задокументований процес оцінювання системи захисту інформації на відповідність встановленим вимогам, стандартам та політикам безпеки.

### Мета аудиту ІБ

- Визначення поточного стану захищеності інформаційних активів організації
- Оцінка відповідності системи управління інформаційною безпекою (СУІБ) встановленим стандартам
- Виявлення вразливостей та загроз
- Надання рекомендацій щодо підвищення рівня захисту

### Завдання аудиту ІБ

- Аналіз політик, процедур та організаційних заходів безпеки
- Перевірка технічних засобів захисту інформації
- Оцінка ризиків інформаційної безпеки
- Перевірка обізнаності персоналу щодо вимог ІБ
- Формування звіту з рекомендаціями

### Принципи аудиту

- **Незалежність** — аудитор не повинен мати конфлікту інтересів з об'єктом аудиту
- **Об'єктивність** — висновки базуються на фактичних доказах
- **Конфіденційність** — захист інформації, отриманої під час аудиту
- **Систематичність** — структурований підхід до збору та аналізу даних
- **Доказовість** — кожен висновок підтверджується зібраними доказами

### Класифікація аудиту ІБ

- **За ініціатором**: внутрішній (проводиться організацією) та зовнішній (незалежною стороною)
- **За обов'язковістю**: обов'язковий (вимагається законодавством) та добровільний
- **За об'єктом**: аудит СУІБ, аудит технічних засобів, аудит процесів, комплексний аудит
- **За періодичністю**: плановий (за графіком) та позаплановий (за потребою)

## Стандарти аудиту інформаційної безпеки

### ISO/IEC 27001

Міжнародний стандарт, що визначає вимоги до **системи управління інформаційною безпекою (СУІБ)**. Основні аспекти:

- Визначення контексту організації та зацікавлених сторін
- Лідерство та політика ІБ
- Планування — оцінка ризиків та обробка ризиків
- Підтримка — ресурси, компетентність, обізнаність, документація
- Операційна діяльність — впровадження заходів управління ризиками
- Оцінка результативності — моніторинг, внутрішній аудит, аналіз з боку керівництва
- Постійне вдосконалення

Додаток A стандарту містить перелік заходів безпеки (controls), згрупованих за категоріями.

### ISO 19011

Стандарт, що надає **настанови щодо аудиту систем управління**. Застосовується до аудиту будь-яких систем управління, включаючи СУІБ.

Ключові елементи:

- **Принципи аудиту**: цілісність, неупередженість, професійна ретельність, конфіденційність, незалежність, підхід на основі доказів, ризик-орієнтований підхід
- **Управління програмою аудиту**: планування, ресурси, виконання, моніторинг
- **Етапи проведення аудиту**: ініціювання, підготовка, проведення, звітування, завершення
- **Компетентність аудиторів**: знання, навички, особисті якості

### Інші релевантні стандарти

- **ISO/IEC 27002** — практичні рекомендації щодо заходів захисту інформації
- **ISO/IEC 27005** — управління ризиками інформаційної безпеки
- **COBIT** — фреймворк управління ІТ-процесами
- **NIST SP 800-53** — каталог заходів безпеки (США)

## Оцінка вразливостей (Vulnerability Assessment)

**Оцінка вразливостей** — систематичний процес ідентифікації, класифікації та пріоритезації вразливостей в інформаційних системах, мережах та програмному забезпеченні.

### Етапи оцінки вразливостей

1. **Визначення обсягу** — які системи та мережі підлягають перевірці
2. **Збір інформації** — інвентаризація активів, версій ПЗ, конфігурацій
3. **Сканування вразливостей** — автоматизоване виявлення відомих вразливостей
4. **Аналіз результатів** — верифікація знайдених вразливостей, відсіювання хибних спрацьовувань
5. **Класифікація та пріоритезація** — оцінка критичності за шкалою CVSS
6. **Звітування** — формування звіту з рекомендаціями щодо усунення

### Типи вразливостей

- **Мережеві** — відкриті порти, незахищені протоколи, неправильна сегментація
- **Системні** — відсутність оновлень, слабкі конфігурації ОС
- **Програмні** — SQL-ін'єкції, XSS, переповнення буфера
- **Організаційні** — слабкі паролі, відсутність політик, недостатня обізнаність персоналу

## Тестування на проникнення (Penetration Testing)

**Тестування на проникнення** — контрольована імітація кібератаки на інформаційну систему з метою виявлення вразливостей, які можуть бути використані зловмисниками.

### Відмінність від оцінки вразливостей

Оцінка вразливостей обмежується виявленням та класифікацією вразливостей. Тестування на проникнення йде далі — передбачає **активну експлуатацію** знайдених вразливостей для підтвердження їх реальної небезпеки.

### Типи тестування на проникнення

- **Black Box** — тестувальник не має попередньої інформації про систему (імітація зовнішнього зловмисника)
- **White Box** — повна інформація про архітектуру, вихідний код, конфігурації
- **Gray Box** — часткова інформація (наприклад, облікові дані звичайного користувача)

### Методологія та етапи

1. **Планування та розвідка (Reconnaissance)** — визначення обсягу, цілей; пасивний та активний збір інформації (OSINT, DNS, WHOIS)
2. **Сканування (Scanning)** — виявлення відкритих портів, сервісів, версій ПЗ; мережева топологія
3. **Отримання доступу (Exploitation)** — експлуатація знайдених вразливостей для проникнення в систему
4. **Підвищення привілеїв (Privilege Escalation)** — спроба отримати адміністративний доступ
5. **Закріплення (Maintaining Access)** — перевірка можливості збереження доступу
6. **Звітування (Reporting)** — детальний звіт з описом знайдених вразливостей, методів експлуатації та рекомендацій

### Стандарти та методології пентесту

- **OWASP Testing Guide** — методологія тестування веб-додатків
- **PTES (Penetration Testing Execution Standard)** — стандарт виконання пентестів
- **OSSTMM** — методологія тестування безпеки з відкритим кодом

## Інструменти аудиту ІБ

### Сканери вразливостей

- **Nessus** — комерційний сканер вразливостей, одне з найпоширеніших рішень
- **OpenVAS** — сканер вразливостей з відкритим кодом
- **Qualys** — хмарна платформа для управління вразливостями

### Інструменти тестування на проникнення

- **Metasploit Framework** — фреймворк для розробки та виконання експлойтів
- **Burp Suite** — платформа для тестування безпеки веб-додатків
- **Nmap** — сканер мережі та портів
- **Kali Linux** — дистрибутив Linux з набором інструментів для пентесту

### Аналіз мережевого трафіку

- **Wireshark** — аналізатор мережевих протоколів
- **tcpdump** — консольний перехоплювач трафіку

### Перевірка паролів

- **John the Ripper** — інструмент для перевірки стійкості паролів
- **Hashcat** — високопродуктивний інструмент для відновлення паролів

## Звітність за результатами аудиту

### Структура звіту з аудиту ІБ

1. **Резюме для керівництва (Executive Summary)** — короткий огляд ключових висновків та ризиків, зрозумілий для нетехнічних зацікавлених сторін
2. **Обсяг та методологія** — опис перевірених систем, використаних методів та інструментів
3. **Виявлені невідповідності та вразливості** — детальний опис кожної знахідки з доказами
4. **Оцінка ризиків** — класифікація за критичністю (критичний, високий, середній, низький)
5. **Рекомендації** — конкретні заходи щодо усунення з визначенням пріоритетів та відповідальних
6. **Додатки** — технічні деталі, логи, скриншоти

### Класифікація знахідок за критичністю

| Рівень | Опис |
|--------|------|
| **Критичний** | Вразливість, яка може бути негайно використана для компрометації системи |
| **Високий** | Суттєва вразливість, що потребує термінового усунення |
| **Середній** | Вразливість з обмеженим впливом або складною експлуатацією |
| **Низький** | Незначна невідповідність, що несе мінімальний ризик |

### Вимоги до звіту

- Точність та об'єктивність — кожен висновок підтверджений доказами
- Зрозумілість — адаптація мови для цільової аудиторії
- Конструктивність — рекомендації повинні бути практичними та реалістичними
- Конфіденційність — звіт містить чутливу інформацію та потребує належного захисту
