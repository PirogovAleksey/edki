# Криптографічний захист інформації

## Основні поняття

**Криптографія** — наука про методи забезпечення конфіденційності, цілісності та автентичності інформації.

**Криптоаналіз** — наука про методи розкриття шифрів без знання ключа.

**Криптологія** = криптографія + криптоаналіз.

Основні сервіси криптографії:
- **Конфіденційність** — захист від несанкціонованого читання
- **Цілісність** — захист від несанкціонованої зміни
- **Автентичність** — підтвердження особи відправника
- **Неспростовність** — неможливість відмовитися від авторства

## Модулярна арифметика (основи для ЄДКІ)

Модулярна арифметика є фундаментом більшості криптографічних алгоритмів. Операція **mod** повертає залишок від ділення.

### Основні властивості

- **(a + b) mod n = ((a mod n) + (b mod n)) mod n**
- **(a * b) mod n = ((a mod n) * (b mod n)) mod n**
- **(a^k) mod n** — можна обчислити ефективно методом швидкого піднесення до степеня (repeated squaring)

### Мала теорема Ферма

Якщо **p** — просте число і **a** не ділиться на **p**, тоді:

**a^(p-1) ≡ 1 (mod p)**

Наслідок: **a^(-1) ≡ a^(p-2) (mod p)** — спосіб знаходження оберненого елемента.

### Функція Ейлера φ(n)

**φ(n)** — кількість чисел від 1 до n, взаємно простих з n.

- Для простого p: **φ(p) = p - 1**
- Для добутку двох простих n = p * q: **φ(n) = (p - 1)(q - 1)**
- Теорема Ейлера: **a^φ(n) ≡ 1 (mod n)** для gcd(a, n) = 1

### Приклади розрахунків

**Приклад 1: 7^10 mod 11**
11 — просте число. За малою теоремою Ферма: 7^10 ≡ 7^(11-1) ≡ 1 (mod 11).
Відповідь: **1**.

**Приклад 2: 3^14 mod 7**
7 — просте число, φ(7) = 6. Тоді 3^14 = 3^(6*2 + 2) = (3^6)^2 * 3^2 ≡ 1^2 * 9 ≡ 2 (mod 7).
Відповідь: **2**.

## Симетричне шифрування

Використовує **один ключ** для шифрування і дешифрування. Швидке, підходить для великих обсягів даних.

### Блокові шифри

Обробляють дані фіксованими блоками:

- **AES** — стандарт шифрування США, блоки 128 біт, ключі 128/192/256 біт
- **DES** — застарілий стандарт, ключ 56 біт, **не є безпечним**
- **3DES** — потрійне застосування DES, ключ 112/168 біт
- **ДСТУ 7624:2014 (Калина)** — український національний стандарт блокового шифрування, блоки 128/256/512 біт
- **ГОСТ 28147-2009** — стандарт блокового шифрування, блок 64 біт, ключ 256 біт

### Режими шифрування блокових шифрів

Визначені стандартом ISO/IEC 10116:2019:

- **ECB** (Electronic Codebook) — кожен блок шифрується незалежно, **не рекомендується** (не забезпечує дифузію)
- **CBC** (Cipher Block Chaining) — кожен блок XOR-ується з попереднім шифроблоком
- **CTR** (Counter) — використовує лічильник, дозволяє паралельне шифрування
- **CFB** (Cipher Feedback) — блоковий шифр працює як потоковий
- **OFB** (Output Feedback) — генерує гамму, не залежить від шифротексту
- **GCM** (Galois/Counter Mode) — забезпечує автентифіковане шифрування (AEAD)

### Потокові шифри

Шифрують дані побітово/побайтово:

- **RC4** — застарілий, **вважається небезпечним**
- **STRUMOK** — український потоковий шифр
- **ChaCha20** — сучасний потоковий шифр (використовується в TLS 1.3)

## Асиметричне шифрування

Використовує **пару ключів**: публічний (для шифрування) та приватний (для дешифрування).

- **RSA** — базується на складності факторизації великих чисел, ключі 2048-4096 біт
- **Ель-Гамаль (ElGamal)** — базується на складності задачі дискретного логарифмування
- **ECC** — еліптичні криві, менші ключі при тій самій стійкості (256-біт ECC ≈ 3072-біт RSA)

### RSA детально

#### Генерація ключів (крок за кроком)

1. Обрати два великих простих числа **p** і **q**
2. Обчислити **n = p * q** — модуль RSA
3. Обчислити **φ(n) = (p - 1)(q - 1)** — функція Ейлера
4. Обрати **e** таке, що **1 < e < φ(n)** і **gcd(e, φ(n)) = 1** (зазвичай e = 65537)
5. Обчислити **d = e^(-1) mod φ(n)** — приватна експонента (за розширеним алгоритмом Евкліда)

**Публічний ключ:** (n, e). **Приватний ключ:** (n, d).

#### Шифрування та дешифрування

- Шифрування: **c = m^e mod n** (де m — повідомлення як число, m < n)
- Дешифрування: **m = c^d mod n**

#### Чому працює RSA

За теоремою Ейлера: **a^φ(n) ≡ 1 (mod n)** для gcd(a, n) = 1.

Оскільки **e * d ≡ 1 (mod φ(n))**, тобто e * d = 1 + k * φ(n), маємо:
**c^d = (m^e)^d = m^(e*d) = m^(1 + k*φ(n)) = m * (m^φ(n))^k ≡ m * 1^k = m (mod n)**.

#### Приклад з малими числами

p = 3, q = 11 → n = 33, φ(n) = 20. Обираємо e = 3 (gcd(3, 20) = 1). Знаходимо d: 3 * d ≡ 1 (mod 20) → d = 7.
Шифрування m = 4: c = 4^3 mod 33 = 64 mod 33 = 31.
Дешифрування: m = 31^7 mod 33 = 4.

### Протокол обміну ключами Діффі-Хеллмана (DH)

Дозволяє двом сторонам узгодити спільний секретний ключ через відкритий канал. Базується на задачі дискретного логарифмування. Варіант на еліптичних кривих — ECDH.

## Хеш-функції

Перетворюють дані довільної довжини у фіксований хеш (дайджест). Властивості:
- **Односторонність** — неможливо відновити вхідні дані
- **Стійкість до колізій** — важко знайти два різних входи з однаковим хешем
- **Лавинний ефект** — мінімальна зміна входу → значна зміна хешу

Основні алгоритми:

- **MD5** — 128 біт, **вважається небезпечним** (знайдено колізії)
- **SHA-1** — 160 біт, **застарілий**
- **SHA-256** — 256 біт, широко використовується (TLS, блокчейн)
- **SHA-3** — стандарт на основі конструкції Keccak
- **Купина (ДСТУ 7564:2014)** — українська хеш-функція, 256/512 біт

## HMAC (Hash-based Message Authentication Code)

**HMAC** — код автентифікації повідомлень на основі хеш-функції. На відміну від простого хешу, HMAC використовує **секретний ключ**, що забезпечує одночасно цілісність та автентичність даних.

### Формула HMAC

**HMAC(K, m) = H((K ⊕ opad) || H((K ⊕ ipad) || m))**

Де:
- **K** — секретний ключ (доповнений до розміру блоку хеш-функції)
- **H** — хеш-функція (SHA-256, SHA-3 тощо)
- **opad** — зовнішнє доповнення (байт 0x5c, повторений до розміру блоку)
- **ipad** — внутрішнє доповнення (байт 0x36, повторений до розміру блоку)
- **||** — конкатенація
- **⊕** — побітове XOR

### Різниця від простого хешу

- Простий хеш H(m) може обчислити будь-хто — він не підтверджує відправника
- HMAC(K, m) вимагає знання секретного ключа K — тільки сторона з ключем може створити коректний HMAC
- Захист від атаки подовження повідомлення (length extension attack), якій піддаються H(K || m)

### Використання HMAC

- **TLS** — перевірка цілісності повідомлень у протоколі захищеного з'єднання
- **API автентифікація** — підпис запитів до API (наприклад, AWS Signature v4)
- **JWT** — підпис токенів (алгоритм HS256 = HMAC-SHA256)
- **TOTP/HOTP** — одноразові паролі (Google Authenticator)

## KDF (Key Derivation Functions)

**Функції виведення ключів** перетворюють пароль або інший секрет у криптографічний ключ. Вони навмисно повільні, щоб ускладнити перебір паролів.

### Чому звичайний хеш не підходить для паролів

- Хеш-функції (SHA-256, MD5) оптимізовані для **швидкості** — мільярди хешів за секунду на GPU
- Атакуючий може перебирати паролі дуже швидко
- KDF додає **ітеративність**, **споживання пам'яті** та **сіль** для протидії

### Основні алгоритми KDF

**PBKDF2** (Password-Based Key Derivation Function 2):
- Ітеративно застосовує HMAC (рекомендовано 600 000+ ітерацій для SHA-256)
- Підтримується стандартом NIST SP 800-132
- Недолік: вразливий до атак на GPU/ASIC через відсутність вимог до пам'яті

**bcrypt:**
- Заснований на шифрі Blowfish з адаптивною складністю
- Обмежений **72 байтами** вхідного пароля (довші паролі обрізаються)
- Вбудована сіль та параметр вартості (cost factor)

**scrypt:**
- **Memory-hard** функція — вимагає великий обсяг оперативної пам'яті
- Параметри: N (CPU/пам'ять), r (розмір блоку), p (паралелізм)
- Ускладнює атаки на спеціалізованому обладнанні (ASIC/FPGA)

**Argon2** (переможець Password Hashing Competition 2015):
- Три варіанти: Argon2d (захист від GPU), Argon2i (захист від side-channel), Argon2id (гібрид)
- Налаштування: обсяг **пам'яті**, **час** (ітерації), кількість **потоків**
- Рекомендований OWASP як основний алгоритм для хешування паролів

## Електронний цифровий підпис (ЕЦП)

Забезпечує **автентичність** та **цілісність**. Створюється приватним ключем, перевіряється публічним.

Алгоритми:
- **DSA** — стандарт цифрового підпису (FIPS 186)
- **ECDSA** — DSA на еліптичних кривих
- **ДСТУ 4145-2002** — український стандарт ЕЦП на еліптичних кривих

### Інфраструктура відкритих ключів (PKI)

- **Сертифікат X.509** — зв'язує публічний ключ з ідентичністю власника
- **Центр сертифікації (CA)** — видає та підписує сертифікати
- **Список відкликаних сертифікатів (CRL)** — перелік недійсних сертифікатів
- **OCSP** — онлайн-протокол перевірки статусу сертифіката

## Perfect Forward Secrecy (PFS)

**Пряма секретність** — властивість протоколу, яка гарантує, що компрометація довгострокового ключа **не розкриває минулі сесії**.

### Принцип роботи

- Для кожної сесії генеруються **ефемерні (тимчасові) ключі** за протоколом DH або ECDH
- Після завершення сесії ефемерні ключі знищуються
- Довгостроковий ключ використовується лише для **автентифікації**, а не для шифрування даних

### Чому це важливо

Без PFS (наприклад, RSA key exchange у TLS 1.2):
- Сервер шифрує pre-master secret публічним ключем RSA
- Якщо приватний ключ RSA скомпрометований — атакуючий може розшифрувати **весь** записаний трафік

З PFS (DHE/ECDHE):
- Кожна сесія використовує унікальну пару ефемерних ключів
- Навіть маючи приватний ключ сервера, неможливо відновити сесійні ключі

### PFS у стандартах

- **TLS 1.3** — PFS є **обов'язковою** (лише ECDHE та DHE)
- **TLS 1.2** — PFS опціональна (залежить від cipher suite)
- **Signal Protocol** — використовує Double Ratchet для PFS на рівні кожного повідомлення

## TLS Handshake детально

**TLS (Transport Layer Security)** — протокол захисту з'єднання, що забезпечує конфіденційність, цілісність та автентифікацію.

### TLS 1.2 Handshake

Повний процес встановлення з'єднання (2-RTT):

1. **ClientHello** — клієнт надсилає: версію TLS, підтримувані cipher suites, випадкове число (Client Random)
2. **ServerHello** — сервер обирає: cipher suite, надсилає випадкове число (Server Random)
3. **Certificate** — сервер надсилає свій сертифікат X.509
4. **ServerKeyExchange** — параметри DH/ECDH (якщо використовується DHE/ECDHE)
5. **ServerHelloDone** — сервер завершив свою частину
6. **ClientKeyExchange** — клієнт надсилає свою частину обміну ключами
7. **ChangeCipherSpec** — перемикання на зашифрований режим
8. **Finished** — підтвердження завершення (MAC від усіх повідомлень handshake)

### TLS 1.3 Handshake

Значно спрощений процес (1-RTT):

1. **ClientHello** — клієнт одразу включає: key_share (параметри ECDHE), підтримувані групи
2. **ServerHello** — сервер обирає параметри, надсилає свій key_share
3. **EncryptedExtensions, Certificate, CertificateVerify, Finished** — все зашифровано вже на цьому етапі
4. **Finished (клієнт)** — підтвердження

### Ключові відмінності TLS 1.3 від 1.2

- **Обов'язковий ECDHE** — видалено RSA key exchange та статичний DH
- **1-RTT** замість 2-RTT (є також режим 0-RTT для повторних з'єднань)
- Видалено небезпечні алгоритми: RC4, DES, 3DES, MD5, SHA-1, CBC-режим
- Спрощено набір cipher suites — лише 5 стандартних

### Cipher Suites (TLS 1.3)

- **TLS_AES_128_GCM_SHA256** — AES-128 у режимі GCM, SHA-256 для HKDF
- **TLS_AES_256_GCM_SHA384** — AES-256 у режимі GCM, SHA-384 для HKDF
- **TLS_CHACHA20_POLY1305_SHA256** — ChaCha20-Poly1305, оптимальний для пристроїв без апаратного AES

## Пост-квантова криптографія

Квантові комп'ютери створюють загрозу для сучасних криптографічних алгоритмів. Підготовка до цієї загрози — одне з головних завдань сучасної криптографії.

### Загроза: алгоритм Шора

**Алгоритм Шора** (1994) дозволяє квантовому комп'ютеру ефективно розв'язувати:
- **Факторизацію** великих чисел — руйнує RSA
- **Задачу дискретного логарифму** — руйнує DH, ECDH, DSA, ECDSA

### Що під загрозою, а що ні

**Під загрозою** (зламуються алгоритмом Шора):
- RSA — будь-яка довжина ключа
- ECC/ECDSA/ECDH — будь-яка крива
- DH/DSA — класичний дискретний логарифм

**Відносно стійкі** (потребують збільшення параметрів через алгоритм Гровера):
- **AES** — подвоїти довжину ключа (AES-256 → ~128-біт квантова стійкість)
- **SHA-256** — збільшити довжину хешу (SHA-256 → ~128-біт квантова стійкість)

### Нові стандарти NIST (2024)

**Для обміну ключами (KEM):**
- **ML-KEM (Kyber)** — FIPS 203, заснований на задачі Module-LWE (решітки)
- Розміри ключів: від 800 до 1568 байт (значно більше за ECDH)

**Для цифрового підпису:**
- **ML-DSA (Dilithium)** — FIPS 204, заснований на задачі Module-LWE
- **SLH-DSA (SPHINCS+)** — FIPS 205, заснований на хеш-функціях (більш консервативний)

### Стратегія переходу

- **Гібридний підхід** — поєднання класичного та пост-квантового алгоритмів (наприклад, ECDH + ML-KEM)
- **Harvest now, decrypt later** — загроза, коли зашифрований трафік записується зараз для дешифрування квантовим комп'ютером у майбутньому
- Рекомендація: починати міграцію вже зараз для критичних систем

## Криптографічні атаки

### Атаки на ключі та паролі

- **Brute force (повний перебір)** — перебір усіх можливих ключів. Складність: 2^n для ключа n біт. Захист: достатня довжина ключа (AES-128+)
- **Dictionary attack (словникова атака)** — перебір слів зі словника та їх комбінацій. Ефективна проти слабких паролів. Захист: складні паролі, KDF
- **Rainbow tables (веселкові таблиці)** — попередньо обчислені таблиці хешів для відомих паролів. Захист: використання **солі** (унікальне випадкове значення для кожного пароля)

### Атаки на хеш-функції

- **Birthday attack (атака днів народження)** — базується на парадоксі днів народження: для хешу довжиною n біт, колізію можна знайти за ~2^(n/2) спроб. Тому SHA-256 має стійкість до колізій ~2^128, а не 2^256

### Атаки на реалізацію (Side-channel attacks)

- **Timing attack** — аналіз часу виконання криптографічних операцій для відновлення ключа. Захист: алгоритми з постійним часом виконання (constant-time)
- **Power analysis** — аналіз споживання електроенергії пристроєм під час криптографічних обчислень. Різновиди: SPA (Simple), DPA (Differential)
- **Cache attack** — аналіз поведінки кешу процесора (Flush+Reload, Prime+Probe)
- **Electromagnetic analysis** — аналіз електромагнітного випромінювання

### Атаки на алгоритми шифрування

- **Known-plaintext attack** — атакуючий знає пару (відкритий текст, шифротекст). Ефективна проти слабких шифрів (наприклад, шифру Цезаря, Enigma)
- **Chosen-plaintext attack** — атакуючий може обирати відкритий текст для шифрування та отримувати відповідний шифротекст. Усі сучасні шифри мають бути стійкими до цієї атаки (IND-CPA)
- **Chosen-ciphertext attack** — атакуючий може обирати шифротекст для дешифрування. Захист: автентифіковане шифрування (AEAD)
- **Padding oracle attack** — експлуатація повідомлень про помилки доповнення у CBC-режимі. Захист: використання AEAD (GCM), а не CBC

## Стеганографія

**Стеганографія** — наука про приховання самого факту передачі інформації. На відміну від криптографії (де повідомлення видно, але не зрозуміло), стеганографія робить повідомлення невидимим.

### Основні поняття

- **Контейнер (носій)** — файл, в який вбудовується приховане повідомлення (зображення, аудіо, відео, текст)
- **Стеганосистема** — сукупність методів та засобів для приховування/вилучення інформації
- **Стеганоключ** — секретний ключ для вбудовування та вилучення прихованого повідомлення
- **Пропускна здатність** — максимальний обсяг прихованих даних без помітного спотворення контейнера

### Методи стеганографії

**За типом контейнера:**
- **Зображення** — заміна найменш значущих біт (LSB), модифікація DCT-коефіцієнтів (JPEG)
- **Аудіо** — LSB, фазове кодування, спектральне приховування
- **Відео** — комбінація методів для зображень + часовий вимір
- **Текст** — зміна міжрядкових/міжсловних інтервалів, синоніми, Unicode-символи

**За підходом:**
- **LSB (Least Significant Bit)** — заміна найменш значущих біт пікселів/семплів. Простий, але вразливий до стеганоаналізу
- **Трансформне кодування** — приховування у частотній області (DCT, DWT). Стійкіше до обробки
- **Статистичне** — модифікація статистичних характеристик контейнера

### Цифрові водяні знаки (ЦВЗ)

Спеціальний випадок стеганографії для захисту авторських прав:

- **Видимі ЦВЗ** — логотипи, написи поверх контенту
- **Невидимі ЦВЗ** — вбудовані в дані, не помітні людському оку/вуху
- **Робастні** — стійкі до обробки (стиснення, обрізка, фільтрація)
- **Крихкі (fragile)** — руйнуються при будь-якій модифікації (для перевірки цілісності)

Застосування:
- Захист авторських прав (copyright protection)
- Відстеження витоків (fingerprinting)
- Підтвердження автентичності

### Стеганоаналіз

Методи виявлення прихованих повідомлень:

- **Візуальний аналіз** — пошук артефактів у зображенні
- **Статистичний аналіз** — виявлення аномалій у розподілі LSB (хі-квадрат тест, RS-аналіз)
- **Сигнатурний аналіз** — пошук слідів відомих стеганографічних програм
- **Порівняльний** — порівняння з оригінальним контейнером (якщо доступний)

## Додаткові концепції криптографії

### Принципи Шеннона

Клод Шеннон визначив два фундаментальні принципи побудови стійких шифрів:

- **Confusion (заплутування)** — ускладнення зв'язку між ключем та шифротекстом. Кожен біт шифротексту має залежати від кількох біт ключа. Досягається за допомогою **S-блоків (Substitution boxes)** — таблиць нелінійної заміни
- **Diffusion (розповсюдження)** — розповсюдження впливу одного біта відкритого тексту на весь шифротекст. Зміна одного біта відкритого тексту повинна змінювати приблизно половину бітів шифротексту. Досягається за допомогою **перестановок (permutations)** та лінійних перетворень

### Принцип Керкгоффса

**Принцип Керкгоффса** (1883) — стійкість криптосистеми має залежати **лише від секретності ключа**, а не від секретності алгоритму. Алгоритм може бути повністю відкритим та опублікованим — це не повинно знижувати безпеку системи. Сучасна інтерпретація: відкриті алгоритми проходять публічний аудит та криптоаналіз, що підвищує довіру до їхньої стійкості.

### AES internals — внутрішня структура

Кожен раунд AES складається з чотирьох операцій у визначеному порядку:

1. **SubBytes** — нелінійна побайтова заміна за допомогою S-блоку (забезпечує confusion)
2. **ShiftRows** — циклічний зсув рядків матриці стану (забезпечує diffusion)
3. **MixColumns** — лінійне перетворення стовпців матриці стану (забезпечує diffusion)
4. **AddRoundKey** — побітове XOR з раундовим ключем

**Важливо:** в останньому раунді операція **MixColumns не виконується**.

**Кількість раундів залежно від довжини ключа:**

| Варіант | Довжина ключа | Кількість раундів |
|---------|---------------|-------------------|
| AES-128 | 128 біт | 10 |
| AES-192 | 192 біт | 12 |
| AES-256 | 256 біт | 14 |

### Схема Ель-Гамаля детально

Криптосистема на основі задачі дискретного логарифмування.

**Відкриті параметри:**
- **p** — велике просте число
- **g** — генератор (примітивний корінь за модулем p)

**Генерація ключів:**
- Обрати секретний ключ **x** (випадкове число, 1 < x < p-1)
- Обчислити відкритий ключ **y = g^x mod p**

**Шифрування повідомлення m:**
- Обрати випадковий сесійний ключ **k** (1 < k < p-1)
- Обчислити пару шифротексту:
  - **a = g^k mod p**
  - **b = m · y^k mod p**

**Дешифрування:**
- **m = b · a^(-x) mod p** або еквівалентно **m = b · (a^x)^(-1) mod p**

**Критична вимога:** сесійний ключ **k має бути унікальним** для кожного повідомлення. Повторне використання k дозволяє зловмиснику обчислити секретний ключ.

### Імітозахист

**Імітозахист** — захист від нав'язування неправдивих даних (підроблених повідомлень або модифікації справжніх). Забезпечує цілісність та автентичність даних.

**Реалізація:**
- **MAC (Message Authentication Code)** — код автентифікації повідомлення, обчислений за допомогою секретного ключа
- **HMAC (Hash-based MAC)** — MAC на основі хеш-функції (найпоширеніший метод)
- Відрізняється від ЕЦП тим, що використовує симетричний ключ (обидві сторони мають однаковий секрет)

### Класичні шифри

- **Шифр Цезаря** — моноалфавітний шифр підстановки, кожна літера зміщується на фіксовану кількість позицій в алфавіті. Наприклад, зсув на 3: А→Г, Б→Д. Легко зламується частотним аналізом (26 можливих ключів для латинського алфавіту)
- **Шифр Віженера** — поліалфавітний шифр підстановки, використовує ключове слово для визначення зсуву кожної літери. Кожна літера ключа визначає свій зсув (А=0, Б=1, ...). Якщо ключ складається з одного символу — шифр Віженера вироджується у шифр Цезаря. Зламується методом Касіскі (визначення довжини ключа) та подальшим частотним аналізом
